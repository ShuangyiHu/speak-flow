# SpeakFlow AI

> An AI-powered English debate coaching platform for Chinese L2 learners —  
> built by a multi-agent development team.

SpeakFlow AI helps students who can write English but struggle to speak it. Through structured debate practice, it builds logical thinking, spoken fluency, and pronunciation confidence — skills that Chinese curricula rarely develop.

What makes this project unusual: **the platform was built by an AI agent team**. A CrewAI-powered development crew (architect, backend engineer, code reviewer, test engineer, frontend engineer) designed, implemented, reviewed, and shipped each module via automated GitHub PRs.

---

## The Problem

Chinese students face three compounding challenges in English speaking:

- **Confidence gap** — years of test-prep culture discourage open-ended expression
- **Logic gap** — curricula rarely train structured argumentation (claim → reason → evidence)
- **Pronunciation gap** — critical for _gaokao_ listening/speaking exams, yet rarely drilled in context

SpeakFlow addresses all three through a single interaction: a real-time AI debate partner.

## Architecture

The project has two distinct layers:

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1 — PRODUCT  (runtime, serves users)                 │
│                                                             │
│  User Audio                                                 │
│      → Whisper (transcription)                              │
│      → TurnAnalyzer (argument + pronunciation analysis)     │
│      → CoachPolicyAgent (LangGraph — decides next action)   │
│      → Response Generator (natural debate reply)            │
│      → SessionEvaluator (end-of-session report)             │
└─────────────────────────────────────────────────────────────┘
           ↑ each module built by the layer below

┌─────────────────────────────────────────────────────────────┐
│  LAYER 2 — DEV TEAM  (build-time, generates the codebase)   │
│                                                             │
│  Requirements spec (MODULES dict in dev_team/main.py)       │
│      → Architect Agent    (design spec)                     │
│      → Backend Engineer   (Python implementation)           │
│      → Code Reviewer      (APPROVE / REQUEST_CHANGES)       │
│      → Test Engineer      (pytest suite)                    │
│      → Frontend Engineer  (Gradio prototype)                │
│      → GitHub PR          (auto-merge if APPROVE)           │
└─────────────────────────────────────────────────────────────┘
```

## Getting Started

### 1. Clone and install dependencies

```bash
git clone https://github.com/<your-username>/speakflow.git
cd speakflow
pip install crewai anthropic PyGithub gitpython langchain-anthropic langgraph gradio
```

### 2. Configure environment variables

```bash
cp .env.example .env
```

Edit `.env` with your credentials (see `.env.example` below).

### 3. Run the Dev Team to generate a module

```bash
cd dev_team
python main.py --module turn_analyzer
```

This will:

1. Run the full agent pipeline (design → implement → review → test → UI)
2. Write output files to `dev_team/output/`
3. Create a GitHub PR with the review report as the PR description
4. Auto-merge if the Code Reviewer approves

### 4. Run the Gradio prototype

```bash
cd app
python app.py
```

## Environment Variables

All environment variables live in a single `.env` file at the **project root**. Both the Dev Team and the product runtime read from this file.

Copy `.env.example` to `.env` and fill in your values:

```bash
# .env.example

# ── LLM ───────────────────────────────────────────────────────
OPENAI_API_KEY=your_open_api_key_here

# ── Dev Team: GitHub Integration ──────────────────────────────
# Personal access token — requires 'repo' scope
GITHUB_TOKEN=your_github_token_here
# Format: username/repo-name
GITHUB_REPO=your-username/speakflow

# ── Product: LangSmith Observability (optional but recommended) ─
LANGCHAIN_API_KEY=your_langsmith_key_here
LANGCHAIN_TRACING_V2=true
LANGCHAIN_PROJECT=speakflow

# ── Product: Development flags ────────────────────────────────
# Set to True to skip Montreal Forced Aligner (for local dev without MFA installed)
USE_STUB_MFA=True
```

> `.env` is in `.gitignore` and will never be committed. Only `.env.example` is tracked.

## Development Workflow

Each new product module follows this pattern:

1. **Add module spec** to `dev_team/main.py` in the `MODULES` dict
2. **Run Dev Team**: `python dev_team/main.py --module <name>`
3. **Review the PR** on GitHub (or let it auto-merge if approved)
4. **Iterate**: if `REQUEST_CHANGES`, update the requirements spec and re-run

This creates a traceable history: every module in `src/` has a corresponding PR with design doc, implementation, review comments, and test results.

## Background

This project was built to demonstrate two things on a single resume:

1. **AI product development** — designing a real-time coaching pipeline with multiple specialized agents, audio processing, and structured feedback
2. **AI-driven development** — using a multi-agent system to _build_ the product itself, with traceable GitHub history

The target users are Chinese high school students preparing for _gaokao_ English speaking exams and beyond — students who have ideas but have never been given the space to argue in English.
